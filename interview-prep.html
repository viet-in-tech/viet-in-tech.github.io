Core ML Concepts

ML Fundamentals

Foundation concepts every DS/ML interview will test. These bridge theory and the projects you've already built.

What is the bias-variance tradeoff and how does it affect model selection? ▼

Bias is error from wrong assumptions — a model too simple to capture the true pattern (underfitting). Variance is error from sensitivity to training data — a model that memorizes noise (overfitting).

The tradeoff: reducing bias often increases variance and vice versa. You want the sweet spot where total error (bias² + variance + irreducible noise) is minimized.

Practical example from your work: In the Beta Bank churn project, a shallow Decision Tree had high bias (F1 = 0.5573). A deep Random Forest with no regularization would have high variance. GridSearchCV helped find the balance — max_depth=10, min_samples_leaf=2.

What's the difference between supervised, unsupervised, and reinforcement learning? ▼
  • Supervised: Labeled training data — the model learns a mapping from inputs to known outputs. Your classification and regression projects are supervised.
  • Unsupervised: No labels — the model finds structure (clusters, embeddings, anomalies) on its own. K-means, PCA, autoencoders.
  • Reinforcement: An agent takes actions in an environment and learns from rewards/penalties. Used in robotics, game AI, recommendation systems.

Your TripleTen projects are all supervised. Be ready to explain how you'd approach an unsupervised problem (e.g., customer segmentation with K-means).

What is regularization and when would you use L1 vs L2? ▼

Regularization adds a penalty term to the loss function to prevent overfitting by discouraging large weights.

  • L2 (Ridge): Penalty = sum of squared weights. Shrinks all weights toward zero but rarely to exactly zero. Good when all features matter somewhat.
  • L1 (Lasso): Penalty = sum of absolute weights. Drives some weights to exactly zero — effective feature selection. Use when you suspect many features are irrelevant.
  • Elastic Net: Combination of both. Useful when you have many correlated features.

Tree-based methods (Random Forest, XGBoost) don't use L1/L2 regularization directly — they use max_depth, min_samples_leaf, and n_estimators to control complexity.

How does a Random Forest work and why is it robust? ▼

Random Forest builds an ensemble of decision trees, each trained on a bootstrap sample (random rows with replacement) of the data, with each split considering only a random subset of features. Predictions are averaged (regression) or majority-voted (classification).

Why it's robust:

  • Bagging reduces variance — individual trees overfit, but their average does not
  • Feature subsampling decorrelates trees, so errors don't all go in the same direction
  • Handles mixed feature types and missing data better than linear models
  • Out-of-bag (OOB) samples provide a built-in validation estimate

From your projects: Random Forest was your top model in both the Megaline plan recommendation (81.8% accuracy) and Beta Bank churn prediction (F1 = 0.6197, ROC-AUC = 0.8618).

What is cross-validation and why use it instead of a single train/test split? ▼

Cross-validation (CV) splits data into k folds, trains on k-1 folds, and validates on the remaining fold — repeating k times. The final score is averaged across all folds.

Advantages over a single split:

  • Every data point is used for both training and validation — more efficient use of limited data
  • Reduces variance in the performance estimate — one lucky/unlucky split doesn't drive your conclusion
  • Detects overfitting more reliably

In your GridSearchCV: You used 3-fold CV across 108 hyperparameter combinations — the CV score guided hyperparameter selection without touching the held-out test set.

Core ML Concepts

Model Evaluation & Metrics

Choosing the right metric is as important as choosing the right model. This is consistently tested in DS/ML interviews.

When would you use F1 score instead of accuracy? ▼

Use F1 when the classes are imbalanced and both false positives and false negatives have real cost.

Accuracy pitfall: In the Beta Bank churn project, a model predicting "no churn" for every customer would score 79.6% accuracy — but catch zero churners. That's useless.

F1 = 2 × (Precision × Recall) / (Precision + Recall). It's the harmonic mean — it punishes models that sacrifice one for the other.

  • F1 is symmetric — treats false positives and false negatives equally
  • Weighted F1 — accounts for class sizes; useful for multi-class imbalance
  • Use accuracy only when class distribution is balanced and both error types cost the same
What is ROC-AUC and what does a score of 0.86 tell you? ▼

ROC (Receiver Operating Characteristic) plots True Positive Rate vs. False Positive Rate at every decision threshold. AUC (Area Under the Curve) summarizes this into a single number.

Interpretation:

  • 0.5 = random guessing — no discrimination ability
  • 0.7–0.8 = acceptable for many real-world cases
  • 0.8–0.9 = good
  • 0.9+ = excellent (check for data leakage)

Your Beta Bank model scored 0.8618 — meaning it correctly ranks 86.18% of (churner, non-churner) pairs. It genuinely distinguishes between the classes rather than getting lucky on a few predictions.

AUC is threshold-invariant — it measures discriminative power independent of where you set the classification cutoff, making it useful for comparing models before deployment decisions.

What is RMSE and when is it the right metric for regression? ▼

RMSE (Root Mean Square Error) = √(mean of squared residuals). It measures the average magnitude of prediction errors in the same units as the target variable.

When to use: When large errors are especially bad — squaring amplifies outlier errors. The House Price Prediction API uses RMSE as the model quality metric because a prediction that's off by $100,000 is far worse than one off by $1,000.

Compared to MAE: MAE (Mean Absolute Error) treats all error sizes equally. If your business problem penalizes large errors heavily, RMSE is the right choice.

R² (coefficient of determination) is another option — it tells you what fraction of variance the model explains. Both RMSE and R² appear in regression model evaluation.

Explain precision and recall and the tradeoff between them. ▼

Precision = TP / (TP + FP) — of everything predicted positive, how many were actually positive? (Minimize false alarms)

Recall = TP / (TP + FN) — of all actual positives, how many did we catch? (Minimize misses)

The tradeoff: Lowering the decision threshold catches more true positives (↑ recall) but also flags more false positives (↓ precision). Raising it does the opposite.

Business context matters:

  • Cancer screening: Maximize recall — missing a case is catastrophic
  • Spam filter: Maximize precision — false positives (blocking real emails) are costly
  • Churn prediction: Balance both — hence F1. Your model achieved precision = 0.62 and recall = 0.77 on churners, meaning it catches 77% of at-risk customers at the cost of some false alarms.

Class Imbalance

You've worked this problem directly. Use your Beta Bank project as the anchor for every answer here.

What is class imbalance and why is it a problem? ▼

Class imbalance occurs when one class significantly outnumbers another in a classification dataset. The model learns to predict the majority class most of the time — achieving high accuracy but failing on the minority class that often represents the actual business problem.

Your example: Beta Bank's dataset was 79.6% non-churn / 20.4% churn. Baseline models scoring F1 = 0.55 weren't wrong — they were just defaulting to "stayed" most of the time. After applying class_weight='balanced', F1 jumped to 0.63.

Compare class weights, oversampling, and undersampling as imbalance strategies. ▼

You tested all three in Sprint 9. Here's the comparison:

  • Class weights (class_weight='balanced'): Increases the penalty for misclassifying the minority class. No data manipulation — uses all original samples. F1 = 0.6381 in your project. Best choice when you have enough data and want to keep the natural distribution.
  • Oversampling (upsampling): Duplicates minority class samples until both classes are equal. Increases training set size. F1 = 0.6173. Risk: overfitting to duplicated examples (SMOTE generates synthetic samples instead).
  • Undersampling (downsampling): Removes majority class samples. Faster training, smaller dataset. F1 = 0.5986. Risk: discarding potentially useful information.

All three passed F1 ≥ 0.59. Class weights won because it preserved the full training set and produced the highest F1 and ROC-AUC — with one parameter change.

What is SMOTE and when would you use it over simple oversampling? ▼

SMOTE (Synthetic Minority Oversampling Technique) generates synthetic minority class samples by interpolating between existing minority samples and their nearest neighbors — rather than just copying them.

Advantage over simple oversampling: Avoids exact duplicate samples, reducing overfitting risk. The synthetic points lie in the "neighborhood" of real minority examples, adding more realistic variation.

When to use: When the minority class is very small, when simple duplication leads to poor generalization, or when you want to combine oversampling with undersampling (SMOTE + ENN or SMOTE + Tomek Links).

Trade-off: SMOTE can introduce noise if the minority class is heterogeneous or if the feature space is high-dimensional. Class weights are often a safer starting point.

Feature Engineering

Turning raw data into model-ready inputs. Often where real performance gains come from.

What is data leakage and how do you prevent it? ▼

Data leakage occurs when information from outside the training set — usually from the test set or the future — contaminates the model, leading to overly optimistic evaluation metrics that don't hold in production.

Common forms:

  • Train/test leakage: Fitting a scaler (or imputer) on the entire dataset before splitting — the test set's statistics influence the transformer. Fix: fit only on training data, then transform test data separately.
  • Target leakage: Including a feature that was derived from the target or collected after the event being predicted.
  • Lookahead in time series: Using future values to predict the past.

From your Beta Bank project: You fit StandardScaler only on X_train, then applied (not refitted) it to validation and test — the correct approach to prevent leakage.

Why did you use median imputation for missing Tenure values rather than mean? ▼

You ran a Missing At Random (MAR) test first — comparing churn rates between customers with and without Tenure data. The difference was less than 1%, confirming the missingness was random (not informative). That made imputation safe.

Median over mean because:

  • Median is robust to outliers — customers with 10-year tenure pull the mean upward
  • Tenure is roughly uniform in distribution, so median and mean are close anyway
  • Only 9.1% of data affected — minimal distortion regardless of choice

The key insight: you validated the imputation assumption before imputing. That's the habit that separates careful data science from guessing.

When should you use one-hot encoding vs. label encoding vs. target encoding? ▼
  • One-hot encoding: Nominal categories with no inherent order (Geography: France, Germany, Spain). Creates binary columns, one per category. Use when cardinality is low (<10–15 categories). In your Beta Bank project, Geography was one-hot encoded.
  • Label encoding: Only appropriate for ordinal categories (e.g., Low=0, Medium=1, High=2) or when the algorithm is tree-based (trees can find splits that recover the meaning). Dangerous with linear models on nominal data.
  • Target encoding: Replaces each category with the mean of the target variable for that group. High cardinality situations (cities, zip codes). Requires careful cross-validation to avoid leakage.

Statistics & Probability

Foundational to the work you've already done — your Chicago Taxi project used hypothesis testing directly.

What is a p-value and what does it tell you? ▼

The p-value is the probability of observing a result at least as extreme as the one you got, assuming the null hypothesis is true.

It does NOT tell you: the probability that the null hypothesis is true, or the size of the effect.

Common threshold: α = 0.05. If p < 0.05, you reject the null hypothesis — the result is statistically significant at the 5% level.

From your Chicago Taxi project: You tested whether bad weather statistically significantly increases ride duration from the Loop to O'Hare. Result: t = 6.84, p ≈ 0 — far below α = 0.05. You rejected the null hypothesis and confirmed the effect (+20.6% longer rides in bad weather).

What's the difference between Type I and Type II errors? ▼
  • Type I error (False Positive): Rejecting the null hypothesis when it's actually true. Rate = α (significance level). "Seeing an effect that isn't there."
  • Type II error (False Negative): Failing to reject the null when it's actually false. Rate = β. "Missing a real effect."

Power = 1 - β = probability of detecting a real effect. Increasing sample size improves power.

In ML terms: False positive = flagging a loyal customer as a churner (costly call for nothing). False negative = missing a churner who then leaves. Precision/recall/F1 are the ML equivalents of managing these tradeoffs.

What is the Central Limit Theorem and why does it matter in practice? ▼

The CLT states that the sampling distribution of the sample mean approaches a normal distribution as sample size increases — regardless of the underlying population distribution.

Why it matters:

  • Justifies using t-tests and z-tests on non-normal data with large enough samples
  • Underpins confidence intervals and hypothesis testing
  • Explains why averaging predictions (ensemble methods, bagging) reduces variance

Rule of thumb: n ≥ 30 is often sufficient for the CLT to apply. With smaller samples, rely on non-parametric tests (Mann-Whitney U) or bootstrapping.

Data Science

EDA & Data Cleaning

Every project starts here. Your approach to missing data and distribution analysis sets the foundation.

Walk me through your EDA process on a new dataset. ▼

A structured EDA process you can describe:

  1. Shape and types: df.shape, df.dtypes, df.head() — understand the scale and structure
  2. Missing values: df.isnull().sum() — identify gaps and test if missingness is random (MAR test)
  3. Distributions: Histograms and box plots for numerical features, value counts for categoricals
  4. Target analysis: Check class balance (classification) or target distribution (regression)
  5. Correlations: Heatmap, pairplot — identify multicollinearity and feature-target relationships
  6. Outliers: IQR method or Z-score — decide whether to cap, remove, or investigate
  7. Business context: Validate findings against domain knowledge before modeling
How do you handle outliers in a dataset? ▼

There's no universal rule — the right approach depends on why the outlier exists.

  • Data entry errors: Fix or remove (a salary of $10,000,000 when others are ~$60,000)
  • Rare but real events: Keep — they may be the most important cases (fraud detection)
  • Capping (Winsorization): Set values beyond the 1st/99th percentile to those boundaries — preserves the row without extreme values distorting the model
  • Log transformation: Compresses right-skewed distributions (income, house prices) — makes outliers less extreme without removing them

Tree-based models are relatively robust to outliers. Linear models are sensitive — preprocessing matters more there.

SQL & Data Wrangling

Common in DS interviews alongside Python — your Chicago Taxi project used SQL to pull source data.

What's the difference between INNER JOIN, LEFT JOIN, and FULL OUTER JOIN? ▼
  • INNER JOIN: Returns only rows where the key exists in both tables. Most common — only matching records.
  • LEFT JOIN: Returns all rows from the left table, matched rows from the right. Non-matching right rows get NULL. Use when you want all records from one table regardless of match.
  • FULL OUTER JOIN: Returns all rows from both tables. NULLs where no match. Less common — useful for auditing or comparing two datasets.

Analogy: INNER = intersection; LEFT = everything on the left plus overlap; FULL OUTER = everything from both sides.

When would you use GROUP BY with HAVING vs. WHERE? ▼

WHERE filters rows before aggregation. HAVING filters after aggregation.

SELECT company_name, COUNT(*) as trip_count
FROM trips
WHERE weather = 'bad'           -- filters before aggregation
GROUP BY company_name
HAVING COUNT(*) > 1000;         -- filters after aggregation

Use HAVING when your filter condition involves an aggregate function (COUNT, SUM, AVG). Use WHERE for row-level conditions on base columns.

What are window functions and when do you use them? ▼

Window functions perform calculations across a set of rows related to the current row, without collapsing them into a single aggregated output (unlike GROUP BY).

SELECT name, salary,
  AVG(salary) OVER (PARTITION BY department) as dept_avg,
  RANK()      OVER (PARTITION BY department ORDER BY salary DESC) as dept_rank
FROM employees;

Common window functions: ROW_NUMBER(), RANK(), DENSE_RANK(), LAG(), LEAD(), SUM() OVER, AVG() OVER.

Use when you need running totals, moving averages, ranking within groups, or comparing a row to its neighbors — without losing row-level detail.

Model Deployment & APIs

Your most recent project (House Price API) covers this directly. Use it as your primary example.

How would you expose a trained ML model as a REST API? ▼

Your Sprint 11 project is the answer. The pattern:

  1. Serialize the trained model (joblib.dump(model, 'model.pkl'))
  2. Create a FastAPI application with Pydantic request/response schemas
  3. Load the model once at startup using @app.on_event("startup") — never per request
  4. Implement a POST /predict endpoint that accepts validated features, runs inference, and returns a typed response
  5. Add GET /health and GET /model/info endpoints for monitoring
  6. Handle errors with appropriate HTTP status codes (503 when model not loaded, 500 for unexpected failures)

FastAPI auto-generates /docs (Swagger UI) from the Pydantic schemas — interactive documentation for free.

Why is it important to load the model at startup rather than on each request? ▼

Loading a model from disk (deserializing a .pkl file) is expensive — it can take seconds, especially for large ensembles. Loading per request would:

  • Add latency to every prediction — unacceptable in production
  • Saturate I/O under load if many requests arrive simultaneously
  • Consume unnecessary CPU and memory (repeated deserialization)

Loading once at startup stores the model in memory, shared across all requests — a single cost paid once. FastAPI's @app.on_event("startup") is the correct hook for this. The global model variable is then read-only during inference, which is thread-safe.

What is Pydantic and why use it for API input validation? ▼

Pydantic is a Python library that enforces type annotations and constraints at runtime. FastAPI uses it natively to validate request/response bodies.

Benefits in your House Price API:

  • Binary amenity flags (garden, garage, pool, etc.) are bounded to [0, 1] — invalid values are rejected before reaching the model
  • Area must be positive and ≤ 10,000 sq ft — catches physically impossible inputs
  • Coordinates are bounded to valid lat/lng ranges
  • Schemas auto-generate the /docs Swagger UI — no separate documentation needed
  • Response schemas ensure the API contract is enforced on output too

Without Pydantic, you'd write manual validation checks in every endpoint — tedious, error-prone, and not self-documenting.

ML Engineering

ML Pipelines & Production

What separates a notebook from a system. Common in mid-level and senior DS/ML interviews.

What is a machine learning pipeline and why use one? ▼

An ML pipeline chains preprocessing steps and a model estimator into a single object — so transformations and predictions always happen in the correct order, on the correct data.

from sklearn.pipeline import Pipeline
from sklearn.preprocessing import StandardScaler
from sklearn.ensemble import RandomForestClassifier

pipe = Pipeline([
    ('scaler', StandardScaler()),
    ('model',  RandomForestClassifier(class_weight='balanced'))
])
pipe.fit(X_train, y_train)
pipe.predict(X_test)  # scaler.transform applied automatically

Key benefits: Prevents data leakage (scaler fitted only on training data in fit()), simplifies cross-validation, makes the full model serializable as one unit, and is reproducible.

What is model drift and how would you monitor for it? ▼

Model drift occurs when a model's performance degrades in production because the real-world data distribution has changed from what the model was trained on.

Two types:

  • Data drift (covariate shift): The input feature distribution changes (e.g., housing prices shift due to market changes)
  • Concept drift: The relationship between features and target changes (e.g., what predicts churn changes after a product redesign)

Monitoring approaches:

  • Track prediction distributions over time — sudden shifts signal drift
  • Monitor feature statistics (mean, standard deviation) vs. training baselines
  • Set up periodic retraining schedules and revalidation against holdout data
  • Use tools like Evidently AI, WhyLogs, or AWS Model Monitor

RAG with Amazon Bedrock Knowledge Bases

Applied Project · AWS Cloud Institute — March 5, 2026. End-to-end RAG pipeline: S3 data source → Titan embeddings → OpenSearch Serverless → Nova Lite → RetrieveAndGenerate API. Core AI/ML engineering concept for enterprise applications.

What is RAG and why is it important for enterprise AI? ▼

RAG (Retrieval Augmented Generation) is a pattern that enhances LLM responses by retrieving relevant context from an external knowledge source and injecting it into the prompt before generation.

Why it matters:

  • LLMs have a knowledge cutoff — they can't answer questions about data created after training, or about proprietary internal documents
  • Fine-tuning is expensive and requires retraining whenever data changes — RAG provides dynamic, up-to-date context without retraining
  • Responses are grounded in source documents with traceable citations — critical for enterprise trust and auditability

From your project: An LLM queried directly about AnyCompany's products returned generic answers. The same query routed through the knowledge base retrieved the actual product specs from the internal documents and generated a grounded, accurate response.

What are vector embeddings and how do they enable semantic search? ▼

Embeddings are dense numerical vectors — arrays of floating-point numbers — that encode the semantic meaning of text. Words or passages with similar meaning have vectors that are close together in high-dimensional space.

How semantic search works:

  • At ingestion: each document chunk is converted to an embedding vector and stored in a vector database
  • At query time: the query is converted to an embedding using the same model
  • The vector database returns the chunks whose embeddings are most similar to the query vector (approximate nearest-neighbor search)
  • Similarity is measured by cosine similarity or dot product — not keyword overlap

In your project: Amazon Titan Text Embeddings v2 converted both document chunks and incoming queries into vectors. OpenSearch Serverless stored and indexed those vectors, returning the most semantically relevant chunks for each query — even when the wording didn't match exactly.

What is a vector database and how does it differ from a traditional database? ▼

A vector database is optimized to store, index, and query high-dimensional embedding vectors using approximate nearest-neighbor (ANN) algorithms.

Key differences from traditional databases:

  • Traditional DB: Exact matches on structured fields (WHERE customer_id = 123). Fast for exact lookups, useless for semantic similarity.
  • Vector DB: Similarity search over dense vectors. "Find the 5 chunks most semantically similar to this query embedding." Inexact by design, but orders of magnitude faster than brute-force comparison.

In your project: OpenSearch Serverless served as the vector store with a bedrock-knowledge-base-default-index index. The same role is played by Pinecone, Weaviate, Qdrant, Milvus, or pgvector in custom RAG stacks — the underlying concept is identical across all of them.

Walk me through the RAG pipeline you built end-to-end. ▼

Situation: AnyCompany's customer support VP needed a POC chat app that could answer questions using 6 internal documents — product descriptions, support tickets, CSAT data, sales records, and an investor summary.

Pipeline steps:

  • Data ingestion: Uploaded 6 files (CSV + TXT) to an S3 bucket via AWS CLI
  • Knowledge base creation: Created an Amazon Bedrock Knowledge Base pointing to the S3 bucket, with Titan Text Embeddings v2 as the embedding model and OpenSearch Serverless as the vector store
  • Sync: Triggered a sync — Bedrock chunked each document, generated embeddings, and wrote the vectors to the OpenSearch index
  • Console testing: Validated RAG responses via the Bedrock test panel — same questions asked with and without the knowledge base to confirm the quality difference
  • API access: Queried the knowledge base programmatically using bedrock-agent-runtime boto3 client and the retrieve_and_generate() method

Result: The system answered AnyCompany-specific questions (preferred communication method per customer, most common support issue type, resolution rates) that an LLM without RAG could not answer — all with source citations pointing back to the S3 documents.

When would you use RAG vs. fine-tuning an LLM? ▼

This is a core architectural decision in applied AI. The right choice depends on what problem you're actually solving.

  • Use RAG when: You need the model to answer questions about specific, frequently updated documents. The knowledge changes often (new support tickets, updated product specs). You need traceable source citations. You want to avoid retraining costs and delays.
  • Use fine-tuning when: You want the model to adopt a specific tone, format, or reasoning style. The task requires specialized domain knowledge baked into the model weights (e.g., medical diagnosis, legal reasoning). Latency constraints prevent injecting large context windows at query time.

The practical reality: Most enterprise AI deployments start with RAG because it's faster, cheaper, and more maintainable than fine-tuning. Fine-tuning is reserved for behavior and style changes, not knowledge injection. Your AnyCompany project is a textbook RAG use case — the documents change, citations matter, and no retraining is needed.

What is the RetrieveAndGenerate API and what does its response structure tell you? ▼

The retrieve_and_generate() method on the bedrock-agent-runtime boto3 client performs the full RAG pipeline in a single API call — retrieval + generation — and returns a structured JSON response.

Key response fields:

  • output.text: The generated answer — what you'd show the user in a chat interface
  • citations[].generatedResponsePart.textResponsePart.text: The specific span of the generated output that was grounded by a retrieved chunk
  • citations[].retrievedReferences[].content.text: The raw chunk of text retrieved from the knowledge base that informed that span
  • citations[].retrievedReferences[].location.s3Location.uri: The S3 URI of the source document — full auditability
  • sessionId: Returned by the API; pass it back in subsequent calls to maintain multi-turn conversation context

The sessionId is what makes follow-up questions like "Were most of those issues resolved?" work — it links the new query to the prior conversation so the model understands the reference.

What happens during a knowledge base sync and why is it required? ▼

A sync processes the raw documents in the S3 bucket into queryable vector embeddings in the following steps:

  • Chunking: Documents are split into smaller text segments (chunks) — typically a few hundred tokens each — to fit within the embedding model's input limit and to maximize retrieval precision
  • Embedding: Each chunk is passed through the embedding model (Titan Text Embeddings v2) to produce a dense vector representation
  • Indexing: The vectors are written to the vector store (OpenSearch Serverless) alongside the original chunk text and source metadata

Why it's required after every data update: The knowledge base doesn't automatically monitor S3 for changes. If you add, update, or delete documents in the bucket, you must trigger a re-sync manually (or via automation) for the vector index to reflect the new state. Until then, queries return results based on the old index.

What is serverless computing and what are its tradeoffs? ▼

Serverless means you write functions that run on demand — the cloud provider manages servers, scaling, and availability. You pay per invocation, not for idle time.

Advantages:

  • No server management — focus on code, not infrastructure
  • Automatic scaling — handles spikes without pre-provisioning
  • Cost efficiency for variable/low-traffic workloads — pay only when code runs

Tradeoffs:

  • Cold starts: First invocation after idle period has latency (~seconds) — problematic for latency-sensitive APIs
  • Execution limits: AWS Lambda has a 15-minute max execution time — not suitable for long-running model training
  • Stateless: No persistent memory between invocations — state must be stored externally (S3, DynamoDB)
  • Vendor lock-in: Services like Bedrock tie you to one cloud provider
What is CORS and why did you need to configure it? ▼

CORS (Cross-Origin Resource Sharing) is a browser security mechanism that blocks JavaScript running on one domain from making requests to a different domain — unless the server explicitly allows it.

In your Serverless AI Application, the frontend (served from one origin) calls the API Gateway endpoint (a different origin). Without CORS headers, the browser refuses the request.

Configuration: You enabled CORS on the API Gateway and returned the appropriate headers (Access-Control-Allow-Origin, Access-Control-Allow-Headers, Access-Control-Allow-Methods) from the Lambda function.

CORS is a browser enforcement mechanism — it doesn't apply to server-to-server calls (like a Python script calling an API). That's why you can test Lambda with curl or Postman without CORS issues.

Tell me about a time you received critical feedback and how you handled it. ▼

Situation: In my House Price Prediction API project (Sprint 11), my first submission had all the core logic working correctly — but my reviewer flagged three specific issues: check_health() returned a raw dictionary instead of a Pydantic model, model_version was assigned the entire metadata dict instead of just the version string, and the currency field lacked a default value.

Action: Rather than treating the feedback as criticism, I treated it as a specification. Each flag pointed to a pattern — consistency, schema enforcement, "pass only what you need." I addressed all three, understood the reasoning behind each change, and resubmitted.

Result: All three revisions were accepted on the second pass. More importantly, I now carry those principles into any API work I do — they've become habits, not just corrections.

Describe a complex problem you broke down into manageable parts. ▼

Example from Beta Bank project:

The churn prediction problem had several compounding issues at once: missing data, class imbalance, and multiple model candidates. Instead of tackling everything at once, I structured it as a sequential investigation:

  1. First, understand the data — shape, missingness, class distribution
  2. Validate the imputation assumption (MAR test) before filling values
  3. Establish baselines with no imbalance correction to understand the baseline problem
  4. Test each imbalance strategy in isolation to measure its isolated contribution
  5. Tune only after the strategy is selected — not before

This is the same skill I used in teaching: you can't teach everything at once. You sequence, you validate understanding at each step, and you build systematically. The habit transfers directly.
