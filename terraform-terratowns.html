01

Overview

This project came out of the ExamPro Terraform Beginner Bootcamp 2023 — a hands-on program that goes well beyond running terraform apply against existing providers. The core challenge: build a custom Terraform provider from scratch in Go that can create, read, update, and delete "homes" on TerraTowns, a real cloud hosting platform used as the bootcamp's deployment target.

Alongside the custom provider, I built a reusable Terraform module (terrahome_aws) that provisions the underlying AWS infrastructure — an S3 bucket for static asset storage and a CloudFront distribution for content delivery — and wired both together in a root main.tf that deploys two live homes: a guide on playing Arcanum in 2023 (a classic RPG rescued by community mods) and a page about making Vietnamese Pho.

Writing a custom provider means implementing the full CRUD lifecycle yourself — in Go — each operation backed by real HTTP calls to the TerraTowns REST API. This is how every provider in the Terraform Registry works under the hood.
02

Architecture

The stack has two independent layers that are wired together through Terraform's dependency graph. The AWS layer (the terrahome_aws module) provisions the hosting infrastructure. The TerraTowns layer (the custom provider) registers each home on the platform, using the CloudFront domain name output from the AWS module as its domain_name argument.

main.tf (Terraform Cloud · org: testtf-organization · workspace: terra-house-1) │ ├── module "terrahome_arcanum" & module "terrahome_pho" │ └── ./modules/terrahome_aws │ ├── aws_s3_bucket (static assets) │ ├── aws_s3_bucket_website_configuration │ ├── aws_s3_object index.html + assets/ │ ├── aws_cloudfront_origin_access_control │ ├── aws_cloudfront_distribution (CDN, PriceClass_200) │ └── terraform_data.invalidate_cache (auto-invalidation) │ └── resource "terratowns_home" "arcanum" & "pho" └── terraform-provider-terratowns (local Go binary) └── HTTP ──► TerraTowns API (town: missingo)

State is managed remotely in Terraform Cloud (organization testtf-organization, workspace terra-house-1), keeping the state file safe and enabling future team collaboration. The development environment runs in Gitpod with automated install scripts in bin/ so the full toolchain — Go compiler, Terraform CLI, AWS CLI — spins up consistently from any fresh workspace.

03

The Custom Provider in Go

The terraform-provider-terratowns binary lives in its own directory and is compiled with go build into a local plugin path so Terraform can resolve it. It registers one resource type — terratowns_home — with five schema fields and implements all four CRUD functions using Go's standard net/http package.

The provider itself is configured with three required inputs: the API endpoint, a bearer token (marked sensitive so it never appears in plan output), and the user_uuid validated against UUID format using the github.com/google/uuid library. These three values are passed into every HTTP request via a *Config struct.

Create

resourceHouseCreate — POST /u/:uuid/homes

Encodes the resource's five fields (name, description, domain_name, town, content_version) into a JSON body and sends a POST to config.Endpoint + "/u/" + config.UserUuid + "/homes". On a 200 response, the returned home UUID is stored with d.SetId() — Terraform uses it for all subsequent operations on this resource.

main.go 
func resourceHouseCreate(ctx context.Context, d *schema.ResourceData, m interface{}) diag.Diagnostics {
    var diags diag.Diagnostics
    config := m.(*Config)

    payload := map[string]interface{}{
        "name":            d.Get("name").(string),
        "description":     d.Get("description").(string),
        "domain_name":     d.Get("domain_name").(string),
        "town":            d.Get("town").(string),
        "content_version": d.Get("content_version").(int),
    }
    payloadBytes, err := json.Marshal(payload)

    req, _ := http.NewRequest("POST",
        config.Endpoint+"/u/"+config.UserUuid+"/homes",
        bytes.NewBuffer(payloadBytes))
    req.Header.Set("Authorization", "Bearer "+config.Token)
    req.Header.Set("Content-Type", "application/json")
    req.Header.Set("Accept", "application/json")

    client := http.Client{}
    resp, err := client.Do(req)
    // ... decode response, extract homeUUID ...
    d.SetId(homeUUID)
    return diags
}

Read

resourceHouseRead — GET /u/:uuid/homes/:home_uuid

Sends a GET to the home's URL and syncs the remote state back into Terraform's state file. Critically, if the API returns a 404, the function calls d.SetId("") to tell Terraform the resource no longer exists — this is the standard pattern for handling out-of-band deletions without crashing the plan.

main.go 
func resourceHouseRead(ctx context.Context, d *schema.ResourceData, m interface{}) diag.Diagnostics {
    config := m.(*Config)
    homeUUID := d.Id()

    req, _ := http.NewRequest("GET",
        config.Endpoint+"/u/"+config.UserUuid+"/homes/"+homeUUID, nil)
    req.Header.Set("Authorization", "Bearer "+config.Token)
    req.Header.Set("Content-Type", "application/json")
    req.Header.Set("Accept", "application/json")

    // If the home was deleted outside Terraform, clear the ID
    if resp.StatusCode == http.StatusNotFound {
        d.SetId("")
        return diags
    }
    // ... decode and set individual fields back into state ...
    return diags
}

Update

resourceHouseUpdate — PUT /u/:uuid/homes/:home_uuid

Sends a PUT with the updated fields. One key detail: domain_name is intentionally excluded from the update payload — the TerraTowns API doesn't allow changing the domain after creation, so sending it would cause an error. Only name, description, town, and content_version are included.

Delete

resourceHouseDelete — DELETE /u/:uuid/homes/:home_uuid

Sends a DELETE request and, on success, calls d.SetId("") to signal to Terraform that the resource is gone. This is the standard pattern — Terraform considers any resource with an empty ID as non-existent and removes it from the state file.

Local Testing

Mock Server (Ruby/Sinatra)

A Ruby/Sinatra app in terratowns_mock_server/ mirrors all four API endpoints locally. This made it possible to test Create → Read → Update → Delete against a real HTTP server without touching the live TerraTowns platform — and to verify that the provider correctly handles each status code before going to production.

04

The terrahome_aws Module

The module lives at ./modules/terrahome_aws and is instantiated twice in main.tf — once for each home — with different public_path and content_version inputs. It provisions five resources and outputs the three values Terraform needs to wire things together: bucket_name, website_endpoint, and domain_name (the CloudFront URL).

ResourceWhat it does
aws_s3_bucket Private bucket tagged with UserUuid — no public access
aws_s3_bucket_website_configuration Enables static website hosting with index.html / error.html routing
aws_s3_object (index + assets) Uploads index.html and any assets/*.{jpg,png,gif} via fileset(); etag-tracked but version-triggered
aws_s3_bucket_policy Locks S3 access to the specific CloudFront distribution only, using Origin Access Control (OAC) and a AWS:SourceArn condition
aws_cloudfront_origin_access_control SigV4 signing, signing_behavior = "always" — every request from CloudFront to S3 is signed
aws_cloudfront_distribution CDN with PriceClass_200, default TTLs (0/3600/86400), IPv6 enabled, default CloudFront certificate
terraform_data.invalidate_cache Provisioner that runs aws cloudfront create-invalidation --paths '/*' whenever content_version changes

The lifecycle block on the S3 objects is worth noting: ignore_changes = [etag] prevents Terraform from re-uploading files just because the etag changed locally, while replace_triggered_by = [terraform_data.content_version.output] forces a re-upload when you intentionally bump the content version. This pattern gives fine-grained control over when deployments actually happen.

resource-storage.tf 
resource "aws_s3_object" "upload_assets" {
  for_each = fileset("${var.public_path}/assets", "*.{jpg,png,gif}")
  bucket   = aws_s3_bucket.website_bucket.bucket
  key      = "assets/${each.key}"
  source   = "${var.public_path}/assets/${each.key}"
  etag     = filemd5("${var.public_path}/assets/${each.key}")

  lifecycle {
    replace_triggered_by = [terraform_data.content_version.output]
    ignore_changes       = [etag]
  }
}

resource "aws_s3_bucket_policy" "bucket_policy" {
  bucket = aws_s3_bucket.website_bucket.bucket
  policy = jsonencode({
    "Version" = "2012-10-17",
    "Statement" = {
      "Sid"    = "AllowCloudFrontServicePrincipalReadOnly",
      "Effect" = "Allow",
      "Principal" = { "Service" = "cloudfront.amazonaws.com" },
      "Action"    = "s3:GetObject",
      "Resource"  = "arn:aws:s3:::${aws_s3_bucket.website_bucket.id}/*",
      "Condition" = {
        "StringEquals" = {
          "AWS:SourceArn" = "arn:aws:cloudfront::${data.aws_caller_identity.current.account_id}:distribution/${aws_cloudfront_distribution.s3_distribution.id}"
        }
      }
    }
  })
}
resource-cdn.tf 
resource "aws_cloudfront_origin_access_control" "default" {
  name                              = "OAC ${aws_s3_bucket.website_bucket.bucket}"
  origin_access_control_origin_type = "s3"
  signing_behavior                  = "always"
  signing_protocol                  = "sigv4"
}

resource "terraform_data" "invalidate_cache" {
  triggers_replace = terraform_data.content_version.output

  provisioner "local-exec" {
    command = <<COMMAND
aws cloudfront create-invalidation \
  --distribution-id ${aws_cloudfront_distribution.s3_distribution.id} \
  --paths '/*'
COMMAND
  }
}

Root Configuration — Two Homes Deployed

The root main.tf instantiates the module and the custom provider resource twice — once for Arcanum (a 2001 RPG that ships with bugs but becomes excellent after community mods) and once for Pho (a Vietnamese cooking guide). The CloudFront domain name from each module instance flows directly into its corresponding terratowns_home resource via Terraform's dependency graph.

main.tf 
terraform {
  cloud {
    organization = "testtf-organization"
    workspaces { name = "terra-house-1" }
  }
  required_providers {
    terratowns = {
      source  = "local.providers/local/terratowns"
      version = "1.0.0"
    }
  }
}

module "terrahome_arcanum" {
  source          = "./modules/terrahome_aws"
  user_uuid       = var.user_uuid
  public_path     = var.arcanum.public_path
  content_version = var.arcanum.content_version
}

resource "terratowns_home" "arcanum" {
  name            = "How to play Arcanum in 2023!"
  description     = "Arcanum shipped with a lot of bugs but with mods it's really fun to play!"
  domain_name     = module.terrahome_arcanum.domain_name
  town            = "missingo"
  content_version = var.arcanum.content_version
}

module "terrahome_pho" {
  source          = "./modules/terrahome_aws"
  user_uuid       = var.user_uuid
  public_path     = var.pho.public_path
  content_version = var.pho.content_version
}

resource "terratowns_home" "pho" {
  name            = "It's Pho me!"
  description     = "How to make your own Pho at home!"
  domain_name     = module.terrahome_pho.domain_name
  town            = "missingo"
  content_version = var.pho.content_version
}
06

Reproducible Dev Environment — Gitpod

The full toolchain is automated through .gitpod.yml and bash scripts in bin/. Opening the repo in Gitpod triggers install scripts for the Terraform CLI (following HashiCorp's updated GPG keyring procedure), the AWS CLI, and Go. The scripts use proper shebangs, chmod permissions, and Gitpod's gp env command to persist sensitive values — AWS credentials, the TerraTowns API token, and user UUID — across workspace restarts without ever committing them to the repo.

This is environment-as-code: the setup is version-controlled, repeatable, and portable across any Gitpod workspace. The same principle applies to production dev containers and CI pipelines.
07

What I Learned

  • How Terraform providers work internally — implementing CRUD in Go with the Plugin SDK made the provider contract concrete: schema registration, the importance of d.SetId("") for 404 handling, and why Read must sync state back from the API rather than trusting local state.
  • The lifecycle block is powerful — combining ignore_changes with replace_triggered_by on terraform_data is a clean pattern for controlling exactly when file uploads and cache invalidations happen.
  • CloudFront + S3 with OAC — Origin Access Control (SigV4 signing) is the current best practice for securing S3-backed CloudFront distributions. The bucket policy's AWS:SourceArn condition locks access to one specific distribution, not all of CloudFront.
  • Go for infrastructure tooling — Go compiles to a single static binary with no runtime dependencies, making it ideal for Terraform provider plugins. The code is straightforward once you understand the diag.Diagnostics return pattern.
  • Remote state & Terraform Cloud — remote state locks prevent concurrent applies from corrupting the state file. This is a non-negotiable for any team environment, and Terraform Cloud makes it free and easy to set up.

Key Takeaway

Almost all Terraform work involves consuming existing providers. Building one from scratch exposed the full contract Terraform expects: a compiled plugin binary, schema-validated inputs, CRUD functions that accurately reflect remote state, and clean teardown on delete. Understanding that contract makes debugging any provider — custom or community — significantly faster.
