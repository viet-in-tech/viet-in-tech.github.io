Building a Custom Terraform Provider:
IaC with Go, AWS & TerraTowns

By Viet Nguyen  ·  Completed 2023  ·  10 min read

← Back to Projects
01

Overview

Most Terraform tutorials stop at using existing providers. This project went a step further: I built a custom Terraform provider from scratch in Go that communicates with the TerraTowns platform — a cloud hosting service used in the ExamPro bootcamp. The provider manages "home" resources, letting Terraform create, read, update, and delete hosted properties through a real HTTP API.

Alongside the provider, I wrote a reusable Terraform module (terrahome_aws) that provisions the underlying AWS infrastructure — S3 for storage and CloudFront for content delivery — and wired everything together with Terraform Cloud for remote state management. Two live homes were deployed: an Arcanum game guide and a Vietnamese Pho cooking reference.

Writing a custom provider means implementing the full CRUD lifecycle in Go — Create, Read, Update, and Delete — each backed by real HTTP calls to the TerraTowns API. This is how Terraform actually works under the hood for every provider in the registry.
02

Architecture

The system has three layers: the Terraform provider (Go binary) that translates HCL resource definitions into API calls, the AWS hosting module that provisions the static-site infrastructure, and the TerraTowns platform that registers and serves the homes publicly.

main.tf (root configuration) │ ├── terrahome_aws module │ ├── aws_s3_bucket (static asset storage) │ ├── aws_cloudfront_distribution (CDN) │ └── aws_s3_bucket_policy (origin access control) │ └── terratowns_home resource └── terraform-provider-terratowns (custom Go binary) └── HTTP ──► TerraTowns API

State is stored remotely in Terraform Cloud, making the configuration portable and safe from local machine loss. The development environment runs in Gitpod with automated setup scripts so the full toolchain — Go, Terraform, AWS CLI — spins up consistently from a fresh workspace.

03

Building the Custom Provider in Go

The terraform-provider-terratowns binary is written in Go using the Terraform Plugin SDK. It registers a single resource type — terratowns_home — and implements all four CRUD operations as HTTP requests against the TerraTowns REST API.

Create

resourceHouseCreate — POST /api/u/:uuid/homes

Serializes the resource's schema fields (name, description, domain name, town, content version) into a JSON body and sends a POST request. On success, Terraform stores the returned UUID as the resource ID — subsequent operations use it for targeting. 

// main.go — abbreviated
func resourceHouseCreate(ctx context.Context, d *schema.ResourceData, m interface{}) diag.Diagnostics {
    client := m.(*Client)
    payload := map[string]interface{}{
        "name":            d.Get("name"),
        "description":     d.Get("description"),
        "domain_name":     d.Get("domain_name"),
        "town":            d.Get("town"),
        "content_version": d.Get("content_version"),
    }
    homeUUID, err := client.CreateHome(payload)
    if err != nil {
        return diag.FromErr(err)
    }
    d.SetId(homeUUID)
    return diags
}

Read / Update / Delete

Full CRUD lifecycle

Read calls GET /api/u/:uuid/homes/:home_uuid and syncs the remote state back into Terraform's state file — critical for detecting drift. Update sends a PUT with the changed fields. Delete sends DELETE and calls d.SetId("") to tell Terraform the resource no longer exists.

Every operation includes authentication via a custom Authorization header carrying the user UUID and bearer token — both passed in from Terraform variables so they are never hardcoded in configuration files.

Local Testing

Mock Server for Development

A Ruby/Sinatra app (terratowns_mock_server) mirrors the TerraTowns API endpoints locally, making it possible to test the provider's CRUD logic without hitting the real platform. This pattern — provider + mock server — is standard practice when building and debugging Terraform providers.

04

AWS Infrastructure Module

The terrahome_aws module is a reusable Terraform module that provisions everything needed to serve a static website from AWS. It is instantiated twice in main.tf — once for each TerraTowns home — with different source content paths.

Resource Purpose
aws_s3_bucket Stores static HTML/CSS/JS assets for each home
aws_s3_bucket_policy Locks bucket access to CloudFront only (OAC)
aws_cloudfront_distribution CDN layer for fast global delivery
aws_cloudfront_origin_access_control Secure S3-to-CloudFront authentication

The module outputs the CloudFront domain name, which is then passed to the terratowns_home resource as the domain_name argument — wiring the two halves of the stack together through Terraform's dependency graph. 

# main.tf — wiring the module output into the custom provider resource
module "terrahome_pho" {
  source          = "./modules/terrahome_aws"
  user_uuid       = var.user_uuid
  public_path     = var.pho.public_path
  content_version = var.pho.content_version
}

resource "terratowns_home" "pho" {
  name            = "How to Make Pho"
  description     = "A guide to preparing authentic Vietnamese Pho"
  domain_name     = module.terrahome_pho.domain_name
  town            = "missingo"
  content_version = var.pho.content_version
}
05

Reproducible Dev Environment with Gitpod

The entire toolchain — Terraform CLI, Go, AWS CLI, and the custom provider build — is automated through .gitpod.yml and shell scripts in bin/. Opening the repo in Gitpod triggers the install scripts, so the environment is always consistent regardless of the local machine.

Sensitive values (AWS credentials, TerraTowns API token, user UUID) are stored as Gitpod environment variables and injected at workspace startup — keeping secrets out of the repository entirely.

This environment-as-code approach mirrors what production teams do with dev containers and CI pipelines: the setup is version-controlled, repeatable, and shareable.
06

What I Learned

  • How Terraform providers actually work — implementing CRUD in Go with the Plugin SDK demystified what happens between terraform apply and the real API call.
  • Terraform module design — writing a reusable module with input variables, local values, and outputs enforces the same separation-of-concerns principles as well-designed application code.
  • AWS CDN architecture — wiring S3 + CloudFront with Origin Access Control (OAC) is the standard pattern for serving static sites securely from AWS.
  • Go for infrastructure tooling — Go's static binary output makes it ideal for CLI tools and provider plugins: one compiled binary, no runtime dependencies.
  • Remote state management — Terraform Cloud state locking prevents concurrent applies from corrupting the state file, a critical concern in team environments.

Key Takeaway

Most IaC work consumes existing providers. Building one revealed the full contract Terraform expects: schema registration, drift detection via Read, and clean teardown via Delete. Understanding that contract makes working with any provider — custom or community — significantly more intuitive.
